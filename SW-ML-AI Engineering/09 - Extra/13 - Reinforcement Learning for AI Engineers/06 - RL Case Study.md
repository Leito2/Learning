# 🏗️ Building an Aligned Agent: End-to-End RLHF Case Study

## Introduction

This case study traces the complete RLHF pipeline from raw base model to aligned agent — not through arcade games, but through a practical scenario: building a customer support agent that is helpful (answers questions accurately), harmless (never generates toxic or dangerous content), and honest (admits uncertainty rather than hallucinating).

This module integrates everything from the previous RL notes: MDP formulation, PPO optimization, reward modeling, and KL-constrained policy updates — and applies them to a single, coherent pipeline. By the end, you will understand every component, every decision point, and every failure mode in a production RLHF deployment.

---

## 1. 🎯 Problem Formulation

### The Goal

Build a customer support agent from a base LLM (Llama-3-8B) that can:
- Answer product questions accurately (helpful)
- Refuse dangerous requests (harmless)
- Say "I don't know" instead of guessing (honest)
- Maintain conversational coherence

### MDP Formulation

| Component | Definition |
|---|---|
| **State $s_t$** | Conversation history (prompt + all previous turns) |
| **Action $a_t$** | Next token from vocabulary $\mathcal{V}$ (~32K tokens) |
| **Policy $\pi_\theta$** | LLM generating token probabilities: $\pi_\theta(a_t \mid s_t)$ |
| **Trajectory $\tau$** | Complete assistant response (sequence of tokens until EOS) |
| **Reward $r$** | Scalar from reward model: $r = r_\phi(\text{prompt}, \text{response})$ |
| **$\gamma = 1.0$** | No discount — every token contributes equally to final response quality |
| **Terminal** | EOS token generated by the policy |

```
┌──────────────────────────────────────────────────────────┐
│              RLHF CUSTOMER SUPPORT AGENT                  │
│                                                          │
│  STATE = "Customer: My account was charged twice.         │
│           Can you refund one charge?"                     │
│                                                          │
│  π_θ outputs token distribution:                         │
│  "I" (0.34), "Of" (0.21), "Let" (0.15), ...             │
│                                                          │
│  Sampled token: "I"                                      │
│  Next state: same conversation + "I"                     │
│                                                          │
│  ... continue generating until EOS ...                   │
│                                                          │
│  Full response: "I understand your frustration. Let me   │
│  check your recent transactions. To process a refund,    │
│  I'll need the date of the duplicate charge."            │
│                                                          │
│  RM scores: helpful=0.92, harmless=0.98, honest=0.95     │
│  Combined reward: 0.3×0.92 + 0.3×0.98 + 0.4×0.95 = 0.95 │
└──────────────────────────────────────────────────────────┘
```

## 3. ⚙️ Stage 1: Supervised Fine-Tuning

### Data Collection

5,000 demonstrations collected from 10 experienced customer support agents. Each demonstration is a (prompt, response) pair covering:

| Category | Examples | Count |
|---|---|---|
| **Account issues** | "I forgot my password", "Change my email" | 1,000 |
| **Billing questions** | "Why was I charged?", "Request refund" | 1,000 |
| **Product questions** | "How do I export data?", "What plan is best?" | 1,500 |
| **Edge cases** | "Delete my account", "Report a bug" | 1,000 |
| **Refusal examples** | "Hack someone's account" → Polite refusal | 500 |

### SFT Training

Standard language modeling loss on the assistant turns only:

```python
from transformers import AutoModelForCausalLM, TrainingArguments, Trainer

model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3-8B")

# SFT: next-token prediction on demonstration data
# Only compute loss on assistant tokens (ignore prompt tokens via masking)

trainer = Trainer(
    model=model,
    args=TrainingArguments(
        output_dir="./sft_model",
        num_train_epochs=3,
        per_device_train_batch_size=4,
        gradient_accumulation_steps=8,
        learning_rate=2e-5,
        warmup_ratio=0.1,
        fp16=True,
    ),
    train_dataset=sft_dataset,
)

trainer.train()
model.save_pretrained("./sft_model")
```

### SFT Baseline Metrics

| Metric | Base Llama-3-8B | After SFT |
|---|---|---|
| **Helpfulness (1-5)** | 2.1 | 3.8 |
| **Format compliance** | 45% | 95% |
| **Refusal rate (dangerous)** | 30% | 60% |
| **Honesty (admits uncertainty)** | 20% | 40% |

SFT improves format and basic helpfulness, but honesty and harmlessness need RL.

## 4. 🎯 Stage 2: Reward Model Training

### Preference Data Collection

For 10,000 prompts, generate 4 responses each from the SFT model (varying temperature: 0.5, 0.7, 0.9, 1.1). Send the 6 pairwise combinations to human labelers (3 labelers per pair).

### Multi-Objective Reward Model

Instead of a single scalar reward, train a multi-head RM:

```python
class MultiHeadRewardModel(nn.Module):
    """Reward model with separate heads for different quality dimensions."""

    def __init__(self, base_model):
        super().__init__()
        self.base = base_model
        self.helpfulness_head = nn.Linear(4096, 1)
        self.harmlessness_head = nn.Linear(4096, 1)
        self.honesty_head = nn.Linear(4096, 1)

    def forward(self, prompt, response):
        features = self.base(prompt, response).last_hidden_state[:, -1, :]
        return {
            "helpfulness": self.helpfulness_head(features),
            "harmlessness": self.harmlessness_head(features),
            "honesty": self.honesty_head(features),
        }
```

### Reward Model Training Results

| Dimension | Pairwise Accuracy | Human Agreement Baseline |
|---|---|---|
| **Helpfulness** | 73% | 72% |
| **Harmlessness** | 78% | 75% |
| **Honesty** | 68% | 65% |
| **Combined (weighted)** | 74% | 71% |

The RM achieves near-human accuracy — sufficient for PPO optimization.

## 5. ⚡ Stage 3: PPO Training

### Training Loop

```python
import torch
from torch.optim import AdamW

# Initialize policy from SFT
policy = load_sft_model("./sft_model")
reference = load_sft_model("./sft_model")  # Frozen
reward_model = load_rm("./reward_model")

optimizer = AdamW(policy.parameters(), lr=1.5e-5)
kl_coef = 0.02
target_kl = 8.0  # nat (natural log units)

# Training prompts (not used in SFT or RM training)
train_prompts = load_prompts("./ppo_prompts.json")

for iteration in range(100):
    batch = sample_batch(train_prompts, batch_size=64)

    # Generate responses from current policy
    responses, old_log_probs = generate_with_logprobs(policy, batch)

    # Compute multi-objective reward
    rm_scores = reward_model(batch, responses)
    reward = (0.3 * rm_scores["helpfulness"] +
              0.3 * rm_scores["harmlessness"] +
              0.4 * rm_scores["honesty"])

    # KL divergence to reference
    with torch.no_grad():
        ref_log_probs = compute_logprobs(reference, batch, responses)
    new_log_probs = compute_logprobs(policy, batch, responses)
    kl = (new_log_probs - ref_log_probs).mean()

    # Adaptive KL coefficient
    if kl > target_kl * 1.5:
        kl_coef *= 2.0
    elif kl < target_kl * 0.5:
        kl_coef *= 0.5

    # Advantage with KL penalty
    advantages = reward - kl_coef * (new_log_probs - ref_log_probs)
    advantages = (advantages - advantages.mean()) / (advantages.std() + 1e-8)

    # PPO update
    ratio = torch.exp(new_log_probs - old_log_probs)
    surr1 = ratio * advantages
    surr2 = torch.clamp(ratio, 0.8, 1.2) * advantages
    policy_loss = -torch.min(surr1, surr2).mean()

    optimizer.zero_grad()
    policy_loss.backward()
    torch.nn.utils.clip_grad_norm_(policy.parameters(), 1.0)
    optimizer.step()

    # Log metrics
    log_metrics({
        "iteration": iteration,
        "mean_reward": reward.mean().item(),
        "kl_divergence": kl.item(),
        "kl_coef": kl_coef,
        "policy_loss": policy_loss.item()
    })

    # Periodic human evaluation every 10 iterations
    if iteration % 10 == 0:
        human_scores = human_evaluation(policy, eval_prompts, n=200)
        log_metrics(human_scores)
```

### Training Dynamics to Monitor

```
Iteration: 0    Mean Reward: 2.1    KL: 0.0    Human Eval: 3.8/5
Iteration: 10   Mean Reward: 3.2    KL: 5.3    Human Eval: 4.1/5  ← Improving
Iteration: 20   Mean Reward: 4.1    KL: 8.2    Human Eval: 4.4/5  ← Good alignment
Iteration: 30   Mean Reward: 5.2    KL: 15.8   Human Eval: 4.2/5  ← RM hacking begins!
Iteration: 40   Mean Reward: 6.8    KL: 28.4   Human Eval: 3.9/5  ← Degraded!
```

The divergence at iteration 30 signals: the RM is being exploited, and the policy is drifting into regions the RM can't properly evaluate. Early stopping at iteration 20-25 would have been optimal.

## 6. 📊 Results and Analysis

### Before and After RLHF

| Metric | Base Model | After SFT | After RLHF (Iter 20) | After RLHF (Iter 40 - Overtrained) |
|---|---|---|---|---|
| **Helpfulness** | 2.1/5 | 3.8/5 | 4.4/5 | 3.9/5 |
| **Harmlessness** | 3.0/5 | 3.5/5 | 4.7/5 | 4.5/5 |
| **Honesty** | 2.0/5 | 2.5/5 | 4.1/5 | 3.2/5 |
| **Coherence** | 4.5/5 | 4.8/5 | 4.6/5 | 3.8/5 |
| **Response diversity** | High | Moderate | Moderate | Low (mode collapse) |

### Key Findings

1. **SFT provides the format baseline** — but alignment requires RL.
2. **KL divergence is the best early warning** for reward hacking.
3. **Adaptive KL control** prevented catastrophic divergence (KL peaked at 15 before correction).
4. **Multi-head RM** enabled targeted improvement: honesty improved most (2.5→4.1) because we weighted it highest.
5. **Overtraining degrades coherence** — the policy optimized for RM scores at the expense of natural language quality.

## 7. 🎯 Production Deployment Considerations

### Inference Pipeline

```
Customer Query → PPO-Aligned Model → Response
                    │
                    ▼
              Safety Filter (rule-based)
              ─ Blocked if:
              ─ Contains PII extraction
              ─ Requests illegal content
              ─ Toxicity score > 0.9
                    │
                    ▼
              Customer Receives Response
```

### Monitoring in Production

| Monitor | Alert If | Action |
|---|---|---|
| **KL from SFT in production** | KL > 20 nats | Rollback to SFT, investigate drift |
| **Human feedback rate** | Satisfaction drops 10% | Trigger retraining pipeline |
| **Dangerous response rate** | > 0.01% of responses | Immediately disable RLHF output, fallback to rule-based |
| **Response length distribution** | Shifts > 2σ from SFT | Policy may be exploiting length bias |

### Continuous Improvement Loop

```
Every 2 weeks:
  1. Sample 1,000 recent production conversations
  2. Send to human labelers for preference annotation
  3. Retrain RM on expanded dataset (now includes production data)
  4. Run one PPO iteration from current checkpoint
  5. A/B test against production model (10% traffic)
  6. If human eval win rate > 55%, promote to 100%
```

## 8. 🔍 Debugging RLHF: The 5 Most Common Failures

| Failure | Symptom | Root Cause | Fix |
|---|---|---|---|
| **Reward hacking** | RM scores ↗ but human eval ↘ | RM misspecification | Retrain RM, increase β |
| **Mode collapse** | Same response for many prompts | Low entropy, over-optimization | Add entropy bonus, reduce PPO epochs |
| **KL explosion** | Policy generates gibberish | β too low, RM weak | Increase β, use adaptive controller |
| **No improvement** | Human eval flat despite RM scores rising | RM not capturing real preferences | Improve RM data quality, add labelers |
| **Catastrophic forgetting** | Loses SFT capabilities | PPO overfits to RM training prompts | Mix SFT loss with PPO loss, use prompt filtering |

## 🎯 Key Takeaways

- An MDP formulates LLM alignment: state = conversation history, action = next token, reward = RM score.
- SFT establishes format and baseline behavior; RLHF layers alignment on top.
- Multi-head reward models enable targeted optimization across quality dimensions (helpfulness, harmlessness, honesty).
- Adaptive KL control prevents reward hacking without manual β tuning.
- The optimal stopping point is NOT when RM scores peak — it's when human eval peaks. These diverge as RM hacking begins.
- Production RLHF requires continuous monitoring and periodic retraining — it's a loop, not a one-time optimization.
