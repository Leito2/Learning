## Learning Path Arc
This course bridges AI-assisted development with production-grade discipline. It teaches how to build a "harness" (arnés) — the operational structure around an AI agent — and how to implement Specification-Driven Development (SDD) as a professional workflow.

## Core Concepts (synthesized from 5 video sources)

### 1. Harness Engineering vs SDD
- **Harness Engineering** = discipline of building the environment/ecosystem around AI models to control them. It automates workflows (SDD, TDD, Agile) via agents.
- **SDD** = a specific workflow within Harness Engineering: Spec → Implement → Validate. The spec is the source of truth.
- Harness does NOT remove power from the agent; it gives DIRECTION.

### 2. The 3 Problems Without Harness
1. **Contaminated context** — agent drags conversation, mixes decisions, loses focus
2. **Unpredictable execution** — sometimes explores, sometimes implements, sometimes invents, touches wrong files
3. **Zero traceability** — why did it do that? what was it trying to validate?

### 3. Key Principles
- **Simplicity over complexity** — Vercel D0 removed 80% of specialized tools, replaced with simple Unix tools (grep, cat, ls). Result: 3x speed, 37% less tokens. Paradox: simpler harness = better performance.
- **Context window degradation** — starts around 20-40% fill. Must externalize memory.
- **External memory > context window** — tasks.json, spec files, design docs live in repo, not in chat.
- **Repository IS the harness** — files in repo define the system, not the model.
- **Multi-agent orchestration** — orchestrator spawns specialized subagents with minimal context.
- **Verification is mandatory** — AI must PROVE work is done (tests, browser, tools), not just claim it.

### 4. Agent Roles in SDD
- **Leader / Orchestrator** — coordinates flow, decides phase, does NOT execute code. Like a tech lead.
- **Spec Author** — writes requirements (EARS format), design.md, tasks.md. Stores output in `specs/<task>/`.
- **Implementer** — writes code following spec. Receives ONLY the curated spec, not full chat history.
- **Reviewer** — validates against spec, architecture conventions, tests. Approve or reject.
- Each role has independent external memory (files, not context window).

### 5. SDD Phase Flow
```
Init → Proposal → Spec & Design (parallel) → Tasks → Apply → Verify → Archive
```
- **Init** — calibrate project: stack, test command, conventions, artifact locations
- **Proposal** — high-level idea of what to build
- **Spec** — requirements (EARS notation or user stories with acceptance criteria)
- **Design** — technical details: files to change, classes, functions, patterns
- **Tasks** — actionable steps for implementer
- **Apply** — code implementation
- **Verify** — tests, review, validation
- **Archive** — store artifacts, update memory
- **Strict gating** — cannot skip phases. Human approval at spec stage.

### 6. The 20 Harnesses (from Gentle framework / Alan Buscalas)
1. **SDD Orchestrator Harness** — coordinates, does NOT execute
2. **Delegation Harness** — when to inline vs delegate to subagent
3. **SDD Init Harness** — project calibration (stack, tests, conventions)
4. **Execution Mode Harness** — interactive (human gates) vs automatic
5. **Artifact Store Harness** — where state lives (repo, Engram, hybrid)
6. **Phase DAG Harness** — prevents skipping phases
7. **Artifact Dependency Harness** — required inputs per phase
8. **Result Contract Harness** — structured handoffs between phases (not prose)
9. **Context Sanitization Harness** — cleans context before each phase
10. **Skill Registry Harness** — auto-discovers available skills
11. **Safety Gate Harness** — blocks dangerous operations
12. **Test Harness** — TDD integration (red → green → refactor)
13. **Review Card Harness** — structured review output
14. **Memory Persistence Harness** — decisions, sessions, learnings (Engram)
15. **Model Routing Harness** — different models per phase (cheap for design, strong for implementation)
16. **Tool Minimalism Harness** — prefer simple Unix tools over specialized wrappers
17. **Session Recovery Harness** — resume after crash/context compaction
18. **Human-in-the-Loop Harness** — gates for approval at critical phases
19. **Traceability Harness** — logs every decision, why it was made
20. **Self-Improvement Harness** — harness updates itself when it detects failures

### 7. Agent Loop Architecture (REPL + Inner Loop)
- **Outer loop (REPL)**: Read input → Evaluate → Print → Loop
- **Inner loop (Tool Execution)**: LLM call → decides tool → executes → returns result → loop until done
- Two loops: user-facing loop + model-facing loop
- Go example: 175 lines core + provider abstraction

### 8. Technical Stack
- Core: LLM SDK (Anthropic, OpenAI, local) via provider abstraction
- Tools: bash, read_file, write_file, grep, ls (Unix philosophy)
- Memory: JSON files, spec directories, Engram (if available)
- Orchestration: leader.md agents + subagent prompts
- Optional: MCP tools, browser automation (Puppeteer/Playwright)

### 9. File Structure Alternatives

#### Alternative A: Gentle-Style (Alan Buscalas)
```
.ai/
├── agents/
│   ├── leader.md          # Orchestrator
│   ├── spec-author.md     # Writes specs
│   ├── implementer.md     # Writes code
│   └── reviewer.md        # Validates
├── skills/                # Reusable commands
│   └── ...
├── harnesses/             # Harness definitions
│   └── ...
specs/
├── <task-name>/
│   ├── requirements.md    # EARS format
│   ├── design.md          # Technical details
│   └── tasks.md           # Actionable steps
tasks.json                 # Task list with status
init.sh                    # Environment verification
agents.md                  # Entry point rules
CLAUDE.md                  # Project context
```

#### Alternative B: Cloud Code Native (Fazt Code style)
```
.claude/
├── CLAUDE.md              # Project context + rules
├── skills/                # /test, /build, /deploy
│   └── test/SKILL.md
├── docs/
│   ├── architecture.md    # 00-overview, 01-stack...
│   ├── decisions/           # ADRs
│   └── runbooks/            # How-to guides
├── tools/
│   └── scripts/
src/
tests/
```

#### Alternative C: Minimalist Unix Harness (Vercel-inspired)
```
.harness/
├── agents/
│   ├── leader.md
│   ├── spec.md
│   └── review.md
├── tools.json             # Simple tool definitions
└── config.yaml            # Stack, test command, conventions
specs/
├── 001-feature-a/
│   ├── req.md
│   ├── design.md
│   └── verify.md
tasks.json
init.sh
```

#### Alternative D: ML/AI Production Harness (user's portfolio alignment)
```
.ai-harness/
├── agents/
│   ├── leader.md
│   ├── spec-author.md
│   ├── implementer.md
│   ├── reviewer.md
│   └── deployer.md         # Deploys to staging/prod
├── memory/
│   ├── decisions.json      # Architecture decisions
│   ├── sessions/           # Per-session context
│   └── learnings.md        # Accumulated insights
├── specs/
│   └── <feature>/
│       ├── requirements.md
│       ├── design.md
│       ├── tasks.md
│       └── verification.md
tasks.json
init.sh
agents.md
CLAUDE.md
```
